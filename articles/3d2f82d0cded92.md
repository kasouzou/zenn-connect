最近、LLM（大規模言語モデル）の出力をそのままプログラムで扱いたい場面が増えてるよね。
でも、AIの返答ってたまにフォーマットが崩れたり、いらないプロパティが混ざったりして「ランタイムエラーで死ぬ」のが一番怖い。

そんなときに便利なのが **Zod** を使ったスキーマバリデーション。
今回は TypeScript プロジェクトで LLM のレスポンスを安全に受け取る方法をサクッと紹介するよ。

## なぜ Zod なのか？

LLM に `JSON` モードで返させて、型定義（interface）を当てるだけでも動くことは動く。
でも、それはあくまで「エディタ上の安心」であって、実際に返ってきたデータが正しい保証はないんだよね。

Zod を使うと：

1. **ランタイムで構造をチェック**できる。
2. 型定義を Zod スキーマから逆引き（`z.infer`）できるから、**二重管理が不要**。
3. 万が一構造が違った場合に、どこが悪いかエラーを出してくれる。

## 実装例

例えば、料理のレシピを生成させる AI 機能を想定してみよう。

### 1. スキーマを定義する

まずは、欲しいデータの形を Zod で定義。

```typescript
import { z } from "zod";

const RecipeSchema = z.object({
  title: z.string(),
  ingredients: z.array(z.string()),
  minutes: z.number().describe("調理時間（分）"),
  difficulty: z.enum(["easy", "medium", "hard"])
});

// ZodスキーマからTypeScriptの型を抽出
type Recipe = z.infer<typeof RecipeSchema>;

```

### 2. LLM の出力をパースする

API から返ってきた文字列（JSON）を `safeParse` するのがポイント。

```typescript
async function getRecipeFromAI(prompt: string) {
  // LLM APIを叩く処理（擬似コード）
  const response = await callLLM(prompt); 
  const jsonResponse = JSON.parse(response);

  // ここでバリデーション！
  const result = RecipeSchema.safeParse(jsonResponse);

  if (!result.success) {
    console.error("AIが変なデータを返したよ:", result.error.format());
    throw new Error("Validation Failed");
  }

  // ここでは型が確定した安全なデータが使える
  return result.data;
}

```

## まとめ

LLM とプログラミングの境界線は、どうしても「不確実性」がつきまとう。
そこを Zod という盾で守ってあげるだけで、開発のストレスがグッと減るからおすすめ！

「AI が嘘をつく」のは防げなくても、「AI がプログラムを壊す」のは防げるよ。
